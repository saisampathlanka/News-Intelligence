# Deployment Guide ‚Äî NewsIntel v5

---

## Pre-deployment checklist

Before deploying to any platform:

- [ ] `SECRET_KEY` is a unique 64-char hex string (`python -c "import secrets; print(secrets.token_hex(32))"`)
- [ ] `ALLOWED_ORIGINS` is set to your actual frontend domain (not `*`)
- [ ] `AUTH_MODE=required`
- [ ] `DATABASE_URL` points to PostgreSQL (not SQLite)
- [ ] `DEBUG=false`
- [ ] `.env` is in `.gitignore` ‚Äî never committed
- [ ] `INITIAL_ADMIN_EMAIL` + `INITIAL_ADMIN_PASSWORD` set for first-run (remove after)

## Option 1: Render (Recommended)

**Why:** Free PostgreSQL included, Blueprint deploys everything in one click, auto-restart on crash.

### Steps

**1. Push to GitHub or GitLab**
```bash
git init
git add .
git commit -m "initial"
git remote add origin https://github.com/YOUR_USER/news-intel.git
git push -u origin main
```

**2. Create Blueprint**
- Go to [dashboard.render.com/blueprints](https://dashboard.render.com/blueprints)
- Click **New Blueprint Instance**
- Connect your repo ‚Äî Render detects `render.yaml` automatically
- Click **Apply** ‚Äî creates:
  - `news-intel-api` (Python web service)
  - `news-intel-frontend` (static site)
  - `news-intel-db` (PostgreSQL free instance)

**3. Set security environment variables**

In the `news-intel-api` service ‚Üí **Environment**:

| Key | Value |
|-----|-------|
| `INITIAL_ADMIN_EMAIL` | `admin@yourcompany.com` |
| `INITIAL_ADMIN_PASSWORD` | A strong password |

> `SECRET_KEY` is auto-generated by Render via `generateValue: true`. Never set it manually on free tier ‚Äî let Render manage it.

**4. Wait for deploy (~3 minutes)**

After the green checkmark:
- API: `https://news-intel-api.onrender.com`
- Frontend: `https://news-intel-frontend.onrender.com`
- Docs: `https://news-intel-api.onrender.com/docs`

**5. Create your admin account**

The app auto-seeds the admin account from `INITIAL_ADMIN_EMAIL` / `INITIAL_ADMIN_PASSWORD` on first startup. Then remove those env vars.

```bash
# Verify:
curl https://news-intel-api.onrender.com/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"admin@yourcompany.com","password":"YourPass1!"}'
```

**6. Ingest first articles**

```bash
TOKEN="eyJ..."   # from the login response above

curl -X POST https://news-intel-api.onrender.com/admin/pipeline \
  -H "Authorization: Bearer $TOKEN"
```

### Free tier limitations

| Limitation | Effect | Fix |
|------------|--------|-----|
| Service sleeps after 15min inactivity | First request after sleep takes ~30s | Upgrade to Render Starter ($7/month) |
| 1 CPU | Scheduler + API share one process | Fine for <50 concurrent users |
| PostgreSQL 1GB storage | ~500K articles | Upgrade when approaching limit |

## Option 2: Fly.io

**Why:** Always-on free tier (no sleep), faster cold starts, global regions.

### Steps

**1. Install Fly CLI and login**
```bash
brew install flyctl    # or https://fly.io/docs/hands-on/install-flyctl/
fly auth login
```

**2. Create app and database**
```bash
fly launch --no-deploy --name news-intel
fly postgres create --name news-intel-db --region ord
fly postgres attach news-intel-db   # auto-sets DATABASE_URL secret
```

**3. Set secrets**
```bash
fly secrets set \
  SECRET_KEY=$(python -c "import secrets; print(secrets.token_hex(32))") \
  AUTH_MODE=required \
  ALLOWED_ORIGINS=https://news-intel.fly.dev \
  INITIAL_ADMIN_EMAIL=admin@example.com \
  INITIAL_ADMIN_PASSWORD=YourStrongPass1!
```

**4. Deploy**
```bash
fly deploy
```

**5. Check logs**
```bash
fly logs
fly status
```

**6. Open app**
```bash
fly open /docs
```

### Fly configuration notes

- `fly.toml` already contains all required `[env]` variables
- Add `NEWSAPI_KEY` etc. via `fly secrets set` (not `fly.toml`)
- Scale: `fly scale vm shared-cpu-2x --memory 1024` if NLP processing is slow
- Region: change `primary_region` in `fly.toml` to the closest to your users

## Option 3: Railway

**Why:** Git-push deploys, generous free tier, simple PostgreSQL plugin.

### Steps

**1. Install Railway CLI**
```bash
npm install -g @railway/cli
railway login
```

**2. Create project**
```bash
railway init
railway add --plugin postgresql   # auto-sets DATABASE_URL
```

**3. Set environment variables**
```bash
railway variables set SECRET_KEY=$(python -c "import secrets; print(secrets.token_hex(32))")
railway variables set AUTH_MODE=required
railway variables set ALLOWED_ORIGINS=https://your-app.railway.app
railway variables set INITIAL_ADMIN_EMAIL=admin@example.com
railway variables set INITIAL_ADMIN_PASSWORD=YourPass1!
```

**4. Deploy**
```bash
railway up
```

Railway detects `.python-version` (3.11.9) and uses Nixpacks with the build command from `railway.toml`.

## Option 4: Docker (self-hosted / VPS)

### Local development

```bash
cp .env.example .env
# Edit .env: set SECRET_KEY, INITIAL_ADMIN_EMAIL, INITIAL_ADMIN_PASSWORD

docker-compose up -d
docker-compose logs -f api
```

Access: `http://localhost:8000`

### Production on a VPS (Ubuntu 22.04)

**1. Install Docker**
```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
```

**2. Clone and configure**
```bash
git clone https://github.com/YOUR_USER/news-intel.git
cd news-intel
cp .env.example .env
# Edit .env with production values
```

**3. Use production compose override**

Create `docker-compose.prod.yml`:
```yaml
version: "3.9"
services:
  api:
    command: uvicorn backend.main:app --host 0.0.0.0 --port 8000 --workers 2
    volumes: []   # no hot reload in production
    environment:
      ENV: production
      DEBUG: "false"
  db:
    restart: always
    ports: []     # don't expose DB port publicly
```

**4. Start with production override**
```bash
docker-compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

**5. Nginx reverse proxy (optional)**

```nginx
server {
    server_name api.yourdomain.com;
    location / {
        proxy_pass http://localhost:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

## Running database migrations

All platforms automatically run `Base.metadata.create_all()` at startup (creates tables if missing).

For schema changes, run Alembic migrations manually:

```bash
# Apply all pending migrations
alembic upgrade head

# Check current revision
alembic current

# Rollback one step
alembic downgrade -1
```

On Render, run via the Shell tab in the service dashboard.
On Fly.io: `fly ssh console -C "alembic upgrade head"`
On Railway: `railway run alembic upgrade head`

## Health monitoring

All platforms check `GET /health`. The endpoint returns:

```json
{"status": "healthy", "db": "connected", "version": "1.0.0"}
```

If the DB is unreachable, it returns `503 Service Unavailable`.

Set your platform's health check path to `/health` with a 30‚Äì40s start period to allow spaCy model loading.
